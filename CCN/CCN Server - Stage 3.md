## Етап 3 – Защита на DNS и устойчивост

**🎯 Цел**

Да се осигури сигурна, устойчива и предвидима работа на DNS сървъра, като се елиминират възможностите за злоупотреба (open resolver, amplification атаки) и се гарантира контрол върху достъпа и поведението на услугата.

---

### Етап 3, Стъпка 1 – Ограничаване на DNS (no recursion)

**🎯 Превръщане на сървъра в чист авторитетен DNS (без рекурсия)**

```
sudo nano /etc/bind/named.conf.options
```

Добавете или коригирайте секцията `options` така:

```
options {
    directory "/var/cache/bind";

    recursion no;
    allow-recursion { none; };
    allow-query-cache { none; };
    allow-query { any; };

    dnssec-validation auto;
};
```

PASS проверка

```
sudo named-checkconf
```

Очакван резултат: 👉 няма изход (конфигурацията е валидна)

**Презареждане на BIND**

```
sudo systemctl restart bind9
```

Очакван резултат 👉 услугата трябва да е активна без грешки

### Етап 3, Стъпка 4 – Активиране на Rate Limiting (RRL)

**🎯 Ограничаване на DNS отговорите за защита от amplification атаки**

```
sudo nano /etc/bind/named.conf.options
```

Добавете в секцията `options`:

```
rate-limit {
    responses-per-second 5;
    window 5;
};
```

Пример за правилно добавена секция:

```
options {
    directory "/var/cache/bind";

    recursion no;
    allow-recursion { none; };
    allow-query-cache { none; };
    allow-query { any; };

    dnssec-validation auto;

    rate-limit {
        responses-per-second 5;
        window 5;
    };
};
```

👉 Спазвайте точно отстъпите:

  * вътрешността на rate-limit трябва да е подравнена  
  * стандартът за BIND конфигурации трябва да бъде спазен

PASS проверка

```
sudo named-checkconf
```

Очакван резултат: 👉 няма изход (конфигурацията е валидна)

**Презареждане на BIND**

```
sudo systemctl restart bind9
```

Очакван резултат 👉 услугата трябва да е активна без грешки

### 1. РЕЗЮМЕ НА ИЗПЪЛНЕНИЕТО / EXECUTIVE SUMMARY

Всички задачи по Етап 3 за подсигуряване и оптимизация на DNS услугата са изпълнени успешно. Сървърът **CCN** е трансформиран в "Strict Authoritative-Only" възел, елиминирайки рисковете от превръщането му в Open Resolver. Внедрени са защити срещу DNS Amplification и DDoS атаки, а техническите детайли на софтуера са скрити от външни скенери.

Изпълнените дейности включват:
* **Hardening:** Пълно изключване на рекурсията и кеширането за външни клиенти.
* **DDoS Protection:** Активиране на Response Rate Limiting (RRL) за ограничаване на злоумишлени заявки.
* **Obscurity:** Скриване на версията на BIND9 (`version "hidden"`).
* **Redundancy Check:** Потвърдена устойчивост на мрежата чрез синхронизация с NS2.
* **Validation:** Проведени успешни тестове за отказ на рекурсия (`REFUSED`) и коректни отговори за собствената зона.
