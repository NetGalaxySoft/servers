## Етап 4 – Конфигуриране на системата за наблюдение и логически анализ

Етап 4 изгражда система за събиране, структуриране, базов анализ и сигнализация на събития, без да взема управленски решения за мрежата.

---

### Етап 4, Стъпка 1 – Синхронизация на системното време (NTP)

**Кратко обяснение**

Всички сървъри в мрежата трябва да работят с **абсолютно еднакво време**, за да може:

* логовете да се подреждат правилно
* събитията да се сравняват коректно
* анализът да бъде достоверен

В тази стъпка ще активираме и проверим системата за синхронизация на време чрез **systemd-timesyncd**.

**Команда**

```bash
timedatectl set-ntp true
```

Очакван резултат

Командата не връща текст, но активира NTP синхронизацията.

**PASS проверка**

```bash
timedatectl status
```

PASS резултат:

Трябва да видите:

```
System clock synchronized: yes
NTP service: active
```

---

### Етап 4, Стъпка 2 – Дефиниране на централен NTP сървър (CCN като референтен източник)

**Кратко обяснение**

В тази стъпка подготвяме CCN да бъде **референтен източник на време за мрежата**.

Целта е:

* всички сървъри (D1, D2 и др.) да използват **един и същ източник**
* да няма разминавания между логовете
* да се осигури стабилна база за анализ

Ще конфигурираме **systemd-timesyncd** да използва външни надеждни NTP сървъри.

**Команда**

```bash
sudo sed -i 's|^#NTP=.*|NTP=0.pool.ntp.org 1.pool.ntp.org 2.pool.ntp.org 3.pool.ntp.org|' /etc/systemd/timesyncd.conf
```

Очакван резултат: файлът `/etc/systemd/timesyncd.conf` ще съдържа ред:

```
NTP=0.pool.ntp.org 1.pool.ntp.org 2.pool.ntp.org 3.pool.ntp.org
```

**PASS проверка**

```bash
grep ^NTP /etc/systemd/timesyncd.conf
```

PASS резултат

```
NTP=0.pool.ntp.org 1.pool.ntp.org 2.pool.ntp.org 3.pool.ntp.org
```

### Етап 4, Стъпка 14 – Активиране на приемане на логове по мрежата (rsyslog)

**Кратко обяснение**

В тази стъпка настройваме CCN да приема логове от други сървъри чрез rsyslog и отваряме необходимите портове във firewall-а.

Активираме:

* UDP порт **514**
* TCP порт **514**

**Команда**

```bash
# Разкоментиране на модулите за UDP и TCP прием
sudo sed -i 's/^#module(load="imudp")/module(load="imudp")/' /etc/rsyslog.conf
sudo sed -i 's/^#input(type="imudp" port="514")/input(type="imudp" port="514")/' /etc/rsyslog.conf
sudo sed -i 's/^#module(load="imtcp")/module(load="imtcp")/' /etc/rsyslog.conf
sudo sed -i 's/^#input(type="imtcp" port="514")/input(type="imtcp" port="514")/' /etc/rsyslog.conf

# Отваряне на портове
sudo ufw allow 514/udp
sudo ufw allow 514/tcp

# Рестарт
sudo systemctl restart rsyslog
```

Очакван резултат

* rsyslog приема логове по UDP и TCP на порт 514
* портовете са отворени във firewall-а
* услугата е активна

---

**PASS проверка**

```bash
ss -tuln | grep 514
sudo ufw status | grep 514
```

PASS резултат

```text
udp   UNCONN 0 0 0.0.0.0:514 ...
tcp   LISTEN 0 128 0.0.0.0:514 ...
```

и

```text
514/udp                   ALLOW
514/tcp                   ALLOW
```

### Етап 4, Стъпка 16 – Класификация на логовете по нива (severity filtering)

**Кратко обяснение**

В тази стъпка разделяме логовете по важност:

* info (информация)
* warning (предупреждение)
* error (грешка)
* crit (критични събития)

Това е първата реална стъпка към „логически анализ“.

**Команда**

```bash
sudo bash -c 'cat > /etc/rsyslog.d/20-severity.conf << "EOF"
# Критични събития
if ($syslogseverity <= 2) then /var/log/remote/critical.log

# Грешки
if ($syslogseverity == 3) then /var/log/remote/error.log

# Предупреждения
if ($syslogseverity == 4) then /var/log/remote/warning.log

# Информация
if ($syslogseverity >= 5) then /var/log/remote/info.log
EOF'

sudo systemctl restart rsyslog
```

Очакван резултат

В директорията `/var/log/remote/` започват да се създават:

* `critical.log`
* `error.log`
* `warning.log`
* `info.log`

**PASS проверка**

```bash
ls -l /var/log/remote
```

PASS резултат

Трябва да видите някои от следните файлове (не е задължително всички):

```text
critical.log
error.log
warning.log
info.log
```

**⚠️ Важно (много)**

👉 Това НЕ заменя файловете по сървъри
👉 Това е **втори слой (анализен слой)**

Имаш вече:

1. по сървъри → `D1.log`, `D2.log`
2. по важност → `error.log`, `critical.log`

🧠 Какво постигаш с това

Вече можеш да правиш:

* „покажи ми всички грешки в мрежата“
* „покажи ми само критични събития“
* „игнорирай шум“

👉 Това е първият реален анализ

### Етап 4, Стъпка 17 – Откриване на неуспешни SSH входове (failed login)

**Кратко обяснение**

В тази стъпка настройваме CCN да разпознава и отделя събития за **неуспешни SSH входове**, като ограничаваме филтъра само до ``sshd``.

**Команда (на CCN)**

```bash
sudo rm -f /etc/rsyslog.d/30-failed-ssh.conf

sudo bash -c 'cat > /etc/rsyslog.d/05-failed-ssh.conf << "EOF"
if ($programname == "sshd" or $syslogtag contains "sshd") and ($msg contains "Failed password" or $msg contains "authentication failure") then {
    action(type="omfile" file="/var/log/remote/failed-ssh.log")
}
EOF'

sudo touch /var/log/remote/failed-ssh.log
sudo chown syslog:adm /var/log/remote/failed-ssh.log
sudo chmod 640 /var/log/remote/failed-ssh.log
sudo systemctl restart rsyslog
logger -t sshd -p authpriv.info "Failed password for root from 127.0.0.1 port 1234 ssh2"
```

**Генериране на тестово събитие (на D1 или D2)**

```bash
ssh invaliduser@localhost
```

👉 Когато поиска парола:

* въведи произволна грешна парола
* натисни Enter 2–3 пъти
* прекрати с `Ctrl+C`

Очакван резултат

На клиентския сървър се генерират логове за неуспешен SSH вход.

**PASS проверка (на CCN)**

```bash
ls -l /var/log/remote
sudo tail -n 5 /var/log/remote/failed-ssh.log
```

PASS резултат 👉 Трябва да видите:

```text
failed-ssh.log
```

и редове от вида:

```text
Failed password for invalid user invaliduser ...
Failed password for root ...
```

**Уточнение**

```text
Ако няма генерирани неуспешни SSH опити, файлът няма да съществува.
```

### Етап 4, Стъпка 19 - Intelligence Report за SSH агресори

**Цел**

**Инсталиране на инструмент за GeoIP**

👉 GeoLite2 чрез пакетите на Ubuntu

```bash

```

**Инсталиране на WHOIS (за организация/ASN)**

```bash
sudo apt install -y whois
```

**Създаване на intelligence скрипт**

```bash
sudo nano /usr/local/bin/ng-intel-report.sh
```

Постави следния код:

```bash
#!/bin/bash

LOG_DIR="/var/log/remote"
TMP_IPS="$(mktemp)"
trap 'rm -f "$TMP_IPS"' EXIT

echo "=== NETGALAXY INTELLIGENCE REPORT: SSH AGGRESSORS ==="
printf "%-16s | %-6s | %-18s | %-24s | %-30s\n" "IP ADDRESS" "TRIES" "COUNTRY" "ORG" "TARGET SERVERS"
echo "------------------------------------------------------------------------------------------------------------------------"

sudo awk '
/from / {
    ip=""
    for(i=1;i<=NF;i++) {
        if($i=="from") {
            ip=$(i+1)
            break
        }
    }
    if(ip!="") print ip
}
' "$LOG_DIR"/failed-ssh.log | sort | uniq -c | sort -nr | head -n 10 > "$TMP_IPS"

while read -r COUNT IP
do
    COUNTRY="$(geoiplookup "$IP" 2>/dev/null | awk -F: '{print $2}' | xargs)"
    [ -z "$COUNTRY" ] && COUNTRY="N/A"

    ORG="$(whois "$IP" 2>/dev/null | grep -iE 'org-name|orgname|descr|netname' | head -n 1 | cut -d: -f2- | xargs)"
    [ -z "$ORG" ] && ORG="N/A"

    TARGETS="$(
        sudo grep -l "from $IP" "$LOG_DIR"/*.log /var/log/auth.log 2>/dev/null \
        | xargs -r -n1 basename \
        | sed 's/\.log$//; s/auth\.log/CCN/' \
        | grep -vE '^(failed-ssh|info|warning|warn|error|errors|crit|debug|notice)$' \
        | sort -u \
        | paste -sd ',' -
    )"
    [ -z "$TARGETS" ] && TARGETS="N/A"

    printf "%-16s | %-6s | %-18s | %-24s | %-30s\n" "$IP" "$COUNT" "$COUNTRY" "$ORG" "$TARGETS"
done < "$TMP_IPS"

echo "------------------------------------------------------------------------------------------------------------------------"
```

### Етап 4, Стъпка 20 - Конфигуриране на Log Rotation за NetGalaxy

**Цел**

Осигуряване на автоматично управление на логовете в `/var/log/remote/`, така че:

* да не се запълни диска
* да се пази достатъчна история за анализ
* rsyslog да работи коректно след ротация

**Създаване на конфигурация за logrotate**

```bash
sudo bash -c 'cat > /etc/logrotate.d/netgalaxy-remote << "EOF"
/var/log/remote/*.log {
    daily
    rotate 14
    missingok
    notifempty
    compress
    delaycompress
    create 640 syslog adm
    sharedscripts
    postrotate
        /usr/lib/rsyslog/rsyslog-rotate
    endscript
}
EOF'
```
**Проверка на конфигурацията**

```bash
sudo logrotate -d /etc/logrotate.conf
```

PASS резултат

* Няма грешки
* В изхода присъства `/var/log/remote/*.log`

*🧠 Какво прави тази настройка:*

  * ***daily*** 👉 Проверява логовете всеки ден и ги ротира при нужда  
  * ***rotate 14*** 👉 Пази 14 дни история  
  * ***compress + delaycompress*** 👉 Старите логове се архивират, a nай-новият архив остава некомпресиран (за бърз достъп)  
  * ***missingok*** 👉 Не дава грешка ако някой лог файл липсва  
  * ***notifempty*** 👉 Не ротира празни файлове  
  * ***create 640 syslog adm*** 👉 След ротация създава нов файл с `правилен собственик` и `правилни права (сигурност)`  
  * ***sharedscripts*** 👉 postrotate се изпълнява само веднъж, не за всеки файл  
  * ***postrotate → rsyslog-rotate*** 👉 Най-важното - казва на rsyslog:

    > „Спря да пишеш в стария файл → започни в новия“

    Без това `rsyslog` ще продължи да пише в стария (вече ротирания) файл ❌

**🎯 Ключов резултат**

След тази стъпка:

* логовете се въртят автоматично ✔
* не се губят данни ✔
* няма риск от пълен диск ✔
* rsyslog остава синхронизиран ✔
